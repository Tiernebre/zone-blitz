# Incident 003: Drizzle migration journal timestamps caused silent skip of 4 migrations in production

## Summary

For an unknown number of deploys, four Drizzle migrations (`0029`–`0032`)
silently failed to apply in production despite the migrator logging
`"Migrations complete."` on every container start. The missing
`leagues.advance_policy` column caused every `GET /api/leagues` and
`POST /api/leagues` request to return HTTP 500, so no user could view or create
a league on `zoneblitz.app` until the journal was fixed and redeployed.

## Timeline

All times UTC on 2026-04-15.

| Time  | Event                                                                                                                                        |
| ----- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| —     | Migration `0028_jersey_number` was committed with a hand-edited `when` value of `1776260000000` in `server/db/migrations/meta/_journal.json` |
| —     | Migrations `0029`–`0032` were generated by `drizzle-kit generate`, each with a `when` value numerically lower than `0028`'s                  |
| 13:27 | Deploy completed; app container reported `Migrations complete.` but applied zero of the 4 pending migrations                                 |
| 13:38 | User attempted to list leagues — endpoint 500'd on `column "advance_policy" does not exist`                                                  |
| 13:39 | User attempted to create a league named "Test" — same error                                                                                  |
| ~14:0 | User reported prod broken; logs scoured, journal diagnosed as root cause                                                                     |
| ~14:3 | Journal entries for `0029`–`0032` bumped above `1776260000000`; CI check + incident doc added                                                |

## Root Cause

Drizzle's `postgres-js` migrator orders journal entries by their `when` field
and applies only those whose `when` is strictly greater than the `created_at` of
the most-recently-applied row in `drizzle.__drizzle_migrations`. It does **not**
reconcile by hash or by `idx`.

The journal contained:

| idx | tag                         | when              |
| --- | --------------------------- | ----------------- |
| 28  | `0028_jersey_number`        | **1776260000000** |
| 29  | `0029_watery_ulik`          | 1776255740654     |
| 30  | `0030_neat_pet_avengers`    | 1776257254069     |
| 31  | `0031_mysterious_red_ghost` | 1776257462862     |
| 32  | `0032_magenta_dark_phoenix` | 1776258798899     |

After `0028` applied successfully (recording `created_at = 1776260000000` in
`__drizzle_migrations`), every subsequent migration looked strictly older than
the newest applied row, so the migrator filtered them out, applied nothing, and
logged `"Migrations complete."` — exactly as designed, given the bad input.

The poison pill was the round-number `1776260000000` on `0028`, almost certainly
inserted by hand during a merge-conflict resolution on the journal file. Drizzle
regenerates `when` as the real `Date.now()` at generate time (~1.776e12 for
mid-April 2026); a round `1776260000000` doesn't occur naturally.

## Resolution

1. Rewrote `when` for entries 29–32 in `_journal.json` so each is strictly
   greater than `0028`'s `1776260000000`, preserving sequential order:
   `1776260000001`–`1776260000004`.
2. Ported the `check-journal-order` script from the sibling `make-the-pick` repo
   (which already enforces monotonic timestamps) as
   `server/db/migrations/check-journal-order.ts`, along with unit tests and a
   regression assertion against the checked-in journal.
3. Added a `db:check-journal` Deno task and a dedicated
   `Migration Journal Order` CI job that runs it on every PR to `main`.
4. Redeployed; the 4 pending migrations applied cleanly and league endpoints
   recovered.

## Lessons Learned

### What went well

- App logs tagged every failed request with the exact `DrizzleQueryError`
  including SQL, which pointed straight at the missing column on the first
  `docker logs` pass.
- The `prod` skill's read-only posture forced triage before action; no
  speculative `docker restart` or manual DB edits were needed.
- Once identified, the fix was a one-line journal edit plus a guardrail — small
  blast radius.

### What could be improved

- The migrator reporting success while applying zero migrations is silently
  dangerous. We should either log the list of migrations that were actually
  applied on this run, or assert that the on-disk journal matches what the DB
  has recorded.
- Merge conflicts in `_journal.json` are resolved by hand today and there's
  nothing stopping a bad `when` from landing. A CI guard (now added) is the
  minimum bar; a pre-commit hook would shorten the feedback loop further.
- The deploy pipeline has a commit-sha health check (`/api/health.commit`) but
  no post-deploy assertion that the migration count in the container equals the
  count in the DB. A "migrations drift" probe would have caught this before the
  container took real traffic.
