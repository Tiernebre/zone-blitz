# 0021 — Sim calibration harness: NFL bands as the sim's contract

- **Date:** 2026-04-15
- **Status:** Proposed
- **Area:** game simulation — see
  [`../north-star/game-simulation.md#nfl-as-the-benchmark`](../north-star/game-simulation.md#nfl-as-the-benchmark);
  extends
  [`0015-simulation-resolution-model.md`](./0015-simulation-resolution-model.md)
  (which lists a "Sim Tuning Harness" as an explicit follow-up); consumes the
  band artifacts generated by [`../../../data/`](../../../data/)

## Context

ADR 0015 committed the sim to a play-by-play resolution core and made it the
author of the canonical `PlayEvent` stream. It did not commit to _how we know
the sim is correct_. The north-star is explicit that league-wide aggregates must
sit inside historical NFL bands
([`game-simulation.md` — NFL as the benchmark](../north-star/game-simulation.md#nfl-as-the-benchmark))
and that a failing harness means tuning the sim, not loosening the band.

`data/bands/team-game.json` already exists — distribution summaries (mean, sd,
p10/p25/p50/p75/p90, min/max) for 15 per-team-per-game offensive metrics across
2020–2024 regular-season play-by-play data, generated by
`data/R/bands/team-game.R` from `nflfastR`. Today nothing in
`server/features/simulation/` reads this file and no test asserts the sim's
aggregates against it. Spot-checks suggest v1 resolution is _not_ currently
in-band on several metrics (pass rate, YPA variance, sack distribution), but
that claim has no test to prove or refute it.

Four questions need to be answered before a ticket can be written:

1. **What counts as "in-band?"** The band is a distribution, not a single
   target. We need a tolerance policy that's strict enough to catch real
   regressions but loose enough to tolerate legitimate RNG variance.
2. **At what N?** One simulated game proves nothing; ten million is slow. The
   harness needs a committed sample size per assertion.
3. **Where does the harness run?** Local `deno task`, CI, both?
4. **What's the failure contract?** When a metric drifts out of band, what's the
   required response — and what would ever justify moving the band?

## Decision

Build a **sim calibration harness** under
`server/features/simulation/calibration/` that reads `data/bands/*.json` as the
sim's behavioral contract, runs the play-by-play engine headless across a
committed seed sweep, and asserts that the sim's emergent distributions match
each band within a defined tolerance. The harness runs on a **daily schedule**
(not as a per-PR blocking gate) and opens a GitHub Issue when any metric drifts
out of band. It is also runnable locally so authors can self-check before
opening a PR.

Bands are never imported as sim _inputs_ — they remain an external oracle. The
sim reaches the bands by construction (attribute-driven play resolution, per ADR
0015), not by sampling from them. See _Alternatives considered_ for why.

### Tolerance policy

A metric **passes** if, across the seed sweep, the simulated distribution
matches the band on all three of:

1. **Mean:** sim mean falls within
   `band.mean ± k_mean · band.sd / sqrt(N_sim_games)` where `k_mean = 3.0`
   (roughly a 3-sigma gate on the sampling error of the mean — loose enough to
   tolerate RNG, tight enough that a 5% drift in the true mean fails at
   realistic N).
2. **Spread:** sim standard deviation falls within
   `[0.75 · band.sd, 1.25 · band.sd]` (±25%). A sim that's correct on average
   but half as variable as the NFL is still wrong (per the north-star:
   "distributions, not just means").
3. **Tails:** simulated p10 ≥ `band.p10 − 0.5 · band.sd` and simulated p90 ≤
   `band.p90 + 0.5 · band.sd`. Catches the failure mode where the sim matches
   the center but never produces a 400-yard passing game or a 2-turnover
   disaster.

A metric **fails** if any of the three fail. The harness reports _all_ failing
metrics for a run, not just the first — tuning one knob often moves several
metrics, and the tuner needs the full picture.

Tolerance constants (`k_mean`, spread ratio, tail slack) live in one constants
file. Changing them is itself a decision worth a short follow-up ADR — the
tolerance is part of the contract.

### Seed sweep methodology

- **N = 2,688 simulated team-games per run**, matching the N of `team-game.json`
  (2,686 team-games across the 5-season window, rounded up to a power of a
  convenient sweep size). Seeds are derived deterministically from a single
  `CALIBRATION_SEED` constant so runs are byte-reproducible.
- The sweep simulates full games (not synthetic possessions) so downstream
  drive/box-score derivation is exercised end-to-end. Each game contributes two
  team-game samples (home + away).
- Rosters used by the sweep come from a fixed "calibration league" snapshot — a
  generated league seeded once and checked in under
  `server/features/simulation/calibration/fixtures/`. This isolates sim
  calibration from player-generator drift; a generator change is a separate ADR
  and its own calibration run.

### What the harness asserts today

Every metric in `data/bands/team-game.json`: plays, pass_attempts,
rush_attempts, pass_rate, rush_rate, completion_pct, yards_per_attempt,
yards_per_carry, pass_yards, rush_yards, sacks_taken, interceptions,
fumbles_lost, turnovers, penalties. New bands land as new assertions
automatically once a loader for their JSON shape is written — the band JSON is
the schema.

### CI integration

The harness runs on a **daily schedule**, not on every PR. The seed sweep is a
heavy job (thousands of full games) and the expected feedback latency makes it a
poor fit for the per-PR critical path — a PR author waiting 10+ minutes on a
calibration job will either be tempted to merge around it or stop iterating.
Daily cadence still surfaces regressions within a business day of the merge that
caused them, and keeps bisection cheap (one day's worth of merges is a small
search space).

- Separate scheduled workflow (`sim-calibration.yml`), triggered on a daily cron
  plus `workflow_dispatch` for manual runs. Also runnable locally as
  `deno task sim:calibrate` — authors touching sim internals are expected to run
  it before opening a PR, the same way they run tests.
- Does **not** block PR merges. Unit tests, typecheck, and lint remain the
  blocking gates. Calibration is an overnight integration check, not a pre-merge
  gate.
- **Revisit if it gets fast.** If harness latency ever lands comfortably under
  ~2 minutes end-to-end (sim + derivation + assertions), promote it to a per-PR
  blocking job with a paths-filter on `server/features/simulation/**`,
  `server/features/player-generator/**`, and `data/bands/**`. Per-PR is strictly
  better feedback; cadence is a cost compromise, not a principle.
- Coverage thresholds still apply to the harness code itself under the 85% Deno
  rule — the harness is real code, not just scripts.
- On failure, the workflow opens a GitHub issue (or comments on an existing open
  one) with the full failing-metrics report and the SHA range since the last
  green run. The issue is the bisection target; the team treats it as a
  broken-main signal and prioritizes accordingly.

### Failure contract

When the harness fails, the required response — in order — is:

1. **Tune the sim.** The failure is the sim's fault by default (per the
   north-star). Identify which resolution-pipeline knob is responsible (outcome
   weights in `resolve-play.ts`, play-call distributions in `drawOffensiveCall`,
   etc.) and adjust.
2. **Investigate the fixture.** If the calibration league snapshot has drifted
   into an unrealistic configuration (all elite QBs, no replacement-level
   players), regenerate it and land that as its own PR with the new snapshot.
3. **Move the band — rare, requires explicit justification.** Only when the real
   NFL has moved (rule change, era shift) and the shift is documented in the
   band regeneration commit. Moving a band because "the sim can't hit it" is
   explicitly forbidden.

Loosening tolerance constants is _not_ a valid response to failure. Tolerance is
a separate decision reviewed on its own merits.

### Both modes share the harness

Per ADR 0015 "fast mode" is the same engine run headless. The harness runs
against that headless path, which _is_ the play-by-play engine. When a
statistical fast-mode path is ever added (not planned), it would need to clear
the same bars with the same bands — by construction the north-star requires
fast-mode and play-by-play to be statistically indistinguishable.

## Alternatives considered

- **Use bands as sim inputs (sample YPA, sack rate, etc. directly from the band
  distributions)** — rejected. Collapses oracle into implementation: the harness
  would be checking that the sim reproduces numbers it was handed, which proves
  nothing. Also destroys the causal chain (down/distance/personnel/attributes →
  outcome) that ADR 0015 committed to — attributes would become cosmetic
  multipliers on pre-decided outcomes rather than causes of them. Revisit only
  once we have play-level _conditional_ bands (YPA given
  down/distance/coverage), and even then only as priors for unknowns the sim
  can't derive mechanistically.
- **Kolmogorov–Smirnov or other full-distribution test** — rejected for v1. KS
  is more statistically rigorous than mean/sd/tail checks but harder to debug
  when it fails ("the distribution shape is off" is less actionable than "p90
  pass_yards is too low"). The three-gate policy gives tuners a clear signal on
  what drifted. Revisit if the three-gate policy proves too weak in practice.
- **One gate (mean only) for v1, add spread/tails later** — rejected. Mean-only
  is exactly the "variance of 10" failure mode the north-star calls out.
  Shipping the weakest version of the harness first makes it easy to grandfather
  in a sim that passes the weak gate but fails the strong one.
- **Smaller sample size (e.g., 256 sim games)** — rejected. Under-samples the
  tails (interceptions, fumbles_lost have means < 1.0 per team-game; tail
  behavior needs thousands of samples). Harness latency is a CI concern, not a
  correctness concern — optimize the sim, not the sample size.
- **Per-PR blocking job from day one** — rejected for v1. A full seed sweep is
  expected to take several minutes (2,688 games × ADR 0015's ~600 plays/sec
  target + harness overhead). Blocking every PR on that latency pushes authors
  to merge around the job or stop iterating. Daily cadence trades same-PR
  feedback for a one-day bisection window — a cost the team can absorb. If the
  harness ever lands comfortably under ~2 minutes, this alternative becomes the
  right answer; see _CI integration — revisit if it gets fast_.
- **Check bands at the league-season aggregate level only** (one sim season,
  compare totals) — rejected. A full season is ~544 team-games — smaller N than
  the sweep, and the aggregation hides which metric drifted. Team-game is the
  right granularity; the NFL band is already aggregated at team-game.

## Consequences

- **Defines "done" for the v1 sim.** ADR 0015 said _what_ v1 produces; this ADR
  says _when v1 is correct_. Sim-tuning work now has a concrete target.
- **Front-loads sim-tuning cost.** The first calibration run is likely to fail
  on multiple metrics (by the user's own assessment). Each failure is a separate
  tuning investigation. This is the cost of having a real oracle.
- **Accepts a bisection window in exchange for low-latency PR feedback.** Daily
  cadence means a sim regression may land on `main` and not fail until the next
  overnight run. The team accepts that tradeoff; the mitigation is that authors
  touching sim internals are expected to run the harness locally
  (`deno task sim:calibrate`) before opening a PR.
- **Locks in the calibration-league fixture as versioned state.** Regenerating
  it is a decision the harness surfaces — drift in the fixture shows up as drift
  in the sim.
- **Surfaces regressions as tracked Issues, not as red PR checks.** A failing
  scheduled run opens an Issue with the SHA range; fixing it is a first-class
  piece of work, not a bystander comment on someone else's PR.
- **Unblocks follow-up band ADRs.** The four open `sim calibration:` issues
  (#246, #247, #248, #249) produce new band JSON files; once the harness is in
  place, each new band plugs in without further architectural work.
- **Follow-ups not in this ADR:**
  - Harness implementation ticket (via `/file-adr-tickets`)
  - Calibration-league fixture generation + checked-in snapshot
  - Per-metric tuning work — one ticket per metric currently out of band (to be
    filed after the first harness run lands the failure list)
  - Tolerance-constants ADR — if the three-gate policy proves miscalibrated in
    practice
  - Conditional / play-level bands (situational rates, position stat
    concentration, etc.) — existing issues #246–249
