# 0008 — Generate tendency vectors for every coach, not just active coordinators

- **Date:** 2026-04-14
- **Status:** Accepted
- **Area:** schemes — see
  [`../north-star/schemes-and-strategy.md`](../north-star/schemes-and-strategy.md);
  refines
  [`0007-coaching-scheme-implementation.md`](./0007-coaching-scheme-implementation.md);
  related to [`0002-coaches-page.md`](./0002-coaches-page.md)

## Context

0007 introduced per-coordinator tendency vectors and limited v1 application to
OC + DC only. Read literally, that scope also implied _generation_ was limited
to coaches currently holding coordinator chairs — which breaks the north-star's
coaching-tree model.

Every head coach in the league was once a coordinator. Every STC was once a
position coach with a special-teams philosophy. When an OC is promoted to HC, or
an HC is fired and takes a DC job the following year, his scheme identity moves
with him. If tendency vectors only exist for the chair a coach currently
occupies, promotions and demotions trigger a "rebuild the vector" moment that
has no grounding in the fiction and creates jarring week-over-week scheme
shifts.

The coaching-tree, mentorship, and hiring-market systems described in
[`coaches.md`](../north-star/coaches.md) all assume a coach carries an identity
with him. That identity needs to exist in data the moment the coach is
generated, not the moment he happens to be sitting in a coordinator chair.

## Decision

**Every coach generated by the sim has a complete tendency vector,** regardless
of current role. Generation is scoped by coordinator origin, not by active
chair:

1. **Head coaches** get both an offensive and a defensive vector. The vector
   corresponding to the HC's coordinator background (offense or defense, per
   `coaches.specialty`) is fleshed out with conviction; the off-side vector is
   generated with weaker signal — a rough lean inherited from staffs he worked
   on, not a coordinator-quality identity. HC also carries an override vector
   (aggressiveness, 4th-down aggression, situational bias) that was already
   scoped in 0007 but had no generation home.
2. **Special teams coordinators** get a dedicated ST tendency vector covering
   return aggression, onside tendency, fake-punt / fake-FG rate, gunner-role
   usage, and field-position philosophy. 0007 deferred STC content entirely;
   this ADR moves STC generation into scope even though STC application remains
   deferred.
3. **Coordinators (OC, DC)** continue to get their side's vector as in 0007.
4. **Position coaches** do _not_ get scheme tendency vectors. Their data lives
   in a different dimension (development/teaching attributes per `coaches.md`)
   and is out of scope here.

**v1 application scope is unchanged.** Per 0007, `computeFingerprint(staff)`
still reads only from the currently-hired OC and DC. HC vectors, HC overrides,
and STC vectors sit dormant in the data until follow-up ADRs wire them in.
Fingerprint and scheme-fit surfaces on Coaches and Roster pages behave
identically to the 0007 plan.

## Why generate dormant data

The temptation is to only generate what the code reads. That's wrong here for
three reasons:

- **Promotion preserves identity.** When an OC is promoted to HC, his OC vector
  becomes his HC's offensive vector. No rebuild, no discontinuity. Without
  up-front generation, promotion would require synthesizing a vector from
  nothing — which is indistinguishable from replacing the coach.
- **Retread HCs are legible.** A fired HC who takes a DC job next year still has
  his defensive vector on file from his coordinator days. Scouts, reputation
  systems, and coaching-tree views can reason about "what kind of coach is this"
  without waiting for him to accumulate a new resume.
- **The generator is cheap; backfill is expensive.** Generating 17 integers per
  coach at create time costs nothing. Writing a migration later that infers
  tendency vectors for every existing HC and STC across every save file —
  especially after users have been playing for seasons — is a real project.

## Alternatives considered

- **Generate only for the chair the coach currently holds** — rejected. Breaks
  promotion/demotion continuity and creates a backfill problem the first time we
  decide HC or STC tendencies matter.
- **Generate for coordinators; synthesize on promotion** — rejected. "Synthesize
  on promotion" is a polite way of saying "roll new dice," which is the exact
  identity discontinuity we're trying to avoid. A promoted OC should bring _his_
  offensive system with him.
- **Give HC a single override vector and no coordinator-style vectors** —
  rejected. An HC with an offensive background has an opinion on the offense
  that goes beyond "aggressive / conservative" — he has a system he ran for a
  decade. Modeling that as overrides alone loses the texture of the hire.
- **Amend 0007 rather than add 0008** — rejected. 0007 is already on an open PR
  with auto-merge queued; refining generation scope is a coherent standalone
  decision that doesn't invalidate 0007's data model or application plan.

## Consequences

- **Generator scope expands.** `stub-coaches-generator.ts` must produce full
  vectors for HCs (both sides, weighted by background), STCs (ST vector), and
  coordinators (their side). The generator archetype/tree clustering from 0007
  now applies across all three roles.
- **Schema note — not a schema change.** `coach_tendencies` from 0007 already
  holds nullable columns on both sides. An HC simply has both offensive and
  defensive columns populated. STC tendencies require adding ST-specific columns
  (return aggression, fake rate, onside tendency, field-position lean) — a
  one-time schema addition alongside the 0007 migration if this lands first, or
  a follow-up migration if 0007 lands first.
- **Off-side HC vector quality is lower.** The weaker off-side vector is a
  feature, not a bug — an "offensive HC" shouldn't have a DC-quality defensive
  identity. Callers that eventually read HC overrides on the opposite side must
  treat the off-side vector as "lean, not system."
- **No UI change.** The Fingerprint panel and Roster Fit indicator still compose
  only OC + DC per 0005 and 0007. The extra data is invisible to the user until
  a future ADR surfaces it.
- **Follow-ups (unchanged from 0007 except STC note):**
  - Wiring HC overrides into fingerprint composition
  - Wiring STC vector into an ST-specific surface (kick coverage rank, return
    philosophy badges, etc.)
  - Archetype-weight content for position-by-position fit
  - Scheme-transition costs on coordinator changes
  - NPC AI using fingerprint similarity for coaching searches
